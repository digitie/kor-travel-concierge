## 버그(라이브 E2E로 발견)

**`extracted_place_candidates.timestamp_start/end`(및 `video_place_mappings`의 동일 컬럼)가
`varchar(16)`인데, Gemini가 16자보다 긴 타임스탬프 문자열을 반환해 적재 시 truncation 오류가 발생한다.**

```
asyncpg StringDataRightTruncationError: value too long for type character varying(16)
INSERT INTO extracted_place_candidates (... timestamp_start, timestamp_end ...)
```

- **영향**: 한 영상의 후보 하나라도 16자 초과 타임스탬프를 가지면 **그 harvest 작업 전체가 실패**한다
  (이번 키워드 job). 채널/플레이리스트는 우연히 16자 이하라 통과했다.
- **권장 수정**(별도 PR 제안):
  1. 모델·Alembic migration으로 `timestamp_start/end`를 `varchar(32)` 또는 `text`로 확장.
  2. 방어적으로, 후보 적재 직전 타임스탬프 문자열을 정규화/클립(예: 16~32자 상한)해 비정상 포맷을 흡수.
  3. 후처리 한 영상의 적재 실패가 전체 job을 롤백시키지 않도록 per-video 트랜잭션 경계 재검토.

## 관찰

- 지오코딩(VWorld) 정확도가 높았다 — 매핑된 38개 장소가 전부 좌표·주소를 확보했다.
- Gemini POI가 영상 제목/설명/자막 맥락에서 한국 여행지를 잘 추출했다(채널의 전국 산책로, 플레이리스트의
  부산 명소). 영상 설명을 POI 입력으로 더 적극 활용하는 개선은 PR #91로 반영 중이다.
- 이번 실행은 구버전(PR #90/#91 이전) 인스턴스 기준이다. 새 코드(DeepSeek 엔진/사전 프롬프트/설명→POI)와
  위 버그 수정 반영 후 재실행하면 키워드 소스까지 완주할 것으로 본다.
